## 1. 일일 운영 루틴

매일 아래 루틴을 수행합니다. 예상 소요 시간: **약 30~45분/일**

### 🌅 오전 루틴 (10:00)

| 순서 | 작업 | 소요 시간 | 상세 |
|------|------|-----------|------|
| 1 | **신규 가격 제보 검수** | 10분 | 전날 밤 ~ 당일 오전 접수분 일괄 검수 (§3 검수 체크리스트 참조) |
| 2 | **방문 완료 기록 확인** | 5분 | 비정상 방문 기록 플래그 확인 (GPS 이탈, 중복 방문 등) |
| 3 | **버그 제보 확인** | 5분 | 오픈채팅방 + 버그 제보 채널에서 신규 이슈 수집 |
| 4 | **서버 상태 확인** | 2분 | Supabase 대시보드에서 API 응답 시간, 오류율 확인 |

### 🌆 오후 루틴 (18:00)

| 순서 | 작업 | 소요 시간 | 상세 |
|------|------|-----------|------|
| 1 | **오후 가격 제보 검수** | 10분 | 오전 ~ 오후 접수분 일괄 검수 |

## 3. 관리자 검수 루틴

### 3.1 검수 프로세스 흐름

```
[신규 제보 접수]
    ↓
[자동 검증] ← 시스템이 수행
    ├─ 중복 체크
    ├─ 가격 범위 체크
    └─ 증빙 첨부 여부 체크
    ↓
[검수 대기열 진입]
    ↓
[관리자 수동 검수] ← 관리자가 수행
    ├─ 장소 실존 확인 (네이버/카카오맵)
    ├─ 증빙 사진 확인
    ├─ 가격 합리성 검증
    ├─ 중복 재확인
    └─ 최종 판정
    ↓
┌────────┬────────┬────────┐
│ 승인   │ 반려   │ 보류   │
│ → 포인트│ → 사유 │ → 24h  │
│   지급  │   안내 │  재검수│
└────────┴────────┴────────┘
```

### 3.2 검수 우선순위

| 우선순위 | 대상 | 목표 처리 시간 |
|----------|------|----------------|
| 🔴 긴급 | 허위 제보 신고, 자동 플래그 건 | 6시간 이내 |
| 🟡 일반 | 일반 가격 제보 | 24시간 이내 |
| 🟢 후순위 | 가격 변동 제보, 보류 재검수 | 48시간 이내 |

### 3.3 검수 판정 기록

모든 검수 건에 대해 아래 정보를 기록합니다:

```
검수 ID: PR-YYYYMMDD-###
제보자: (사용자 ID)
장소: (장소명)
메뉴: (메뉴명)
제보 가격: (금액)
판정: 승인 / 반려 / 보류
판정 사유: (선택지 또는 자유 서술)
검수자: (관리자 ID)
검수 일시: (YYYY-MM-DD HH:MM)
```

### 3.4 에스컬레이션 기준

아래 상황은 관리자 단독 판단이 아닌 **팀 논의**가 필요합니다:

- 허위 제보 3회 누적 사용자 처분
- 새로운 유형의 부정행위 발견
- 일반가 기준 변경 요청
- 시스템 장애로 인한 대량 오류 발생

## 4. 데이터 품질 관리 루틴

### 4.1 주간 데이터 감사 (매주 월요일)

| # | 점검 항목 | SQL/방법 | 조치 |
|---|-----------|----------|------|
| 1 | **만료 임박 가격** | `updated_at`이 유효기간 30일 전인 건 조회 | "확인 필요" 배지 표시 |
| 2 | **만료된 가격** | 유효기간 초과 건 조회 | "미확인(stale)" 상태 변경 |
| 3 | **미검수 적체** | `status = 'pending'`이 48시간 초과인 건 | 즉시 검수 처리 |
| 4 | **중복 데이터** | 동일 장소+동일 메뉴 중복 승인 건 | 최신 1건만 유지 |
| 5 | **비정상 절약액** | 절약액 상위 10% 건 리스트 | 수동 검증 |
| 6 | **비활성 장소** | 제보가 0건인 장소 중 생성 30일 초과 | 비활성 처리 검토 |

### 4.2 데이터 감사 SQL 예시

**만료 임박 가격 조회:**
```sql
SELECT place_id, menu_name, price, updated_at
FROM price_reports
WHERE status = 'approved'
  AND updated_at < NOW() - INTERVAL '60 days'
ORDER BY updated_at ASC;
```

**미검수 적체 조회:**
```sql
SELECT id, place_id, menu_name, created_at
FROM price_reports
WHERE status = 'pending'
  AND created_at < NOW() - INTERVAL '48 hours'
ORDER BY created_at ASC;
```

**비정상 절약액 조회:**
```sql
SELECT user_id, place_id, savings_amount, created_at
FROM visit_records
WHERE savings_amount > 30000
ORDER BY savings_amount DESC
LIMIT 20;
```

### 4.3 월간 데이터 정리 (매월 1일)

| 작업 | 설명 |
|------|------|
| 만료 가격 비활성화 | 유효기간 + 30일 초과 가격 지도에서 숨김 |
| 카테고리 일반가 재산정 | 신규 데이터 반영하여 일반가 업데이트 |
| 사용자 통계 정리 | 활성/비활성/정지 사용자 현황 정리 |
| 백업 | 전체 데이터 백업 수행 |

### 4.4 데이터 품질 KPI

| 지표 | 산식 | 목표 |
|------|------|------|
| 가격 정확도 | 승인 후 30일 내 수정 없는 비율 | 90%+ |
| 검수 완료율 | 48시간 내 검수 완료 비율 | 95%+ |
| 데이터 신선도 | 유효기간 내 가격 비율 | 80%+ |
| 중복률 | 중복 반려 / 전체 제보 | 10% 이하 |
